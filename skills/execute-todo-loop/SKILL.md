# 执行 TODO 单任务闭环

一次只完成一个 TODO。Obsidian 项目工作区提供目标和长期上下文，真实代码、Git 状态和运行实例提供当前事实。完成后停止，不自动开始下一项。

## 1. 定位项目工作区

1. 读取并遵守用户级和仓库级 `AGENTS.md`、`CLAUDE.md`、开发约束与 Git 规则。
2. 确定仓库根目录；不是 Git 仓库时使用当前工作目录。
3. 读取 `AGENT_PROJECT_WORKFLOW_CONFIG` 指定的配置；未设置时依次尝试 `~/.config/agent-project-workflow/config.toml` 和兼容旧安装的 `~/.codex/project-workflow.toml`。优先使用 `[projects]` 中与仓库绝对路径完全匹配的相对目录；没有映射时使用 `<projects_root>/<仓库名>`，`projects_root` 缺失则按 `Myproject` 处理。
4. 将 `<vault_root>/<解析后的相对目录>/` 作为唯一项目工作区，读取其中的 `TODO.md`、相关 `planning/`、`constraints/` 和 `explanations/`。
5. 配置缺失、路径不可用或映射有歧义时停止并说明；不得回退到仓库创建 `.agent/`，也不得使用仓库中的旧 `.agent/` 作为当前事实源。

不要把 Obsidian 的绝对路径、个人计划或私有上下文写入项目公开文档。API Key、令牌和密码只从系统凭据库或 `~/.config/agent-project-workflow/local/<仓库名>/` 获取，不得写入 Obsidian。

## 2. 调查并选择任务

先完成只读调查：

- 检查当前分支、HEAD、远端跟踪和 dirty worktree。
- 读取 TODO、相关计划、长期约束和验收口径。
- 检查现有 Goal；若已有未完成 Goal，只在它与所选 TODO 一致时继续，否则停止确认。
- 不选择标记为延后、等待条件、暂停或不属于当前阶段的任务。
- 多个任务无法可靠取舍时向用户确认，不擅自选择。
- 读取相关源码、配置、测试、日志或运行状态，理解真实调用链和影响范围。
- 识别未提交改动的归属；不得擅自 `stash`、`reset`、覆盖或混入用户改动。

计划不存在、关键验收口径缺失或任务仍处于延后状态时，不进入实现。

## 3. 核对计划与代码

把差异分为三类：

- **实现缺口**：目标仍成立但实现缺失，继续完成。
- **非实质漂移**：文件、命名或内部结构变化，但目标与边界仍成立；按真实代码调整路径，并修正文档中已经失真的事实。
- **实质冲突**：计划依赖的架构、数据事实源、产品决策、安全边界或验收条件已经不成立；修改代码前停止并说明证据与影响。

不要把普通实现缺口误判为冲突，也不要替用户决定实质冲突。

## 4. 锁定 Goal 与实时步骤

- 用户显式调用本 Skill 时，先检查现有 Goal，再为所选 TODO 创建唯一 Goal。
- Goal 只描述本次任务目标，不覆盖整个 TODO 列表。
- 只有用户明确给出 token budget 时才设置预算。
- 用实时 Plan 维护少量当前步骤，不把执行流水复制回长期计划。
- Goal 工具不可用时，说明限制并询问是否允许继续。

## 5. 分支与实现

始终以仓库 Git 规则为准。仓库要求临时分支时必须遵守；无法判断时默认使用单目标临时分支。

实现时：

1. 定位现有模块、事实源、公共 helper、测试夹具和相邻实现。
2. 明确修改文件、失败路径、兼容边界、回滚方式和验证方法。
3. 只做 Goal 所需的最小充分改动，不混入无关重构、样式或功能。
4. 分小步实现，在每个风险边界后运行 focused tests。
5. 计划外缺陷只有直接阻止 Goal 时才纳入，否则记录为新的 TODO 候选。

未经用户明确授权，不推送、不创建 PR、标签、Release，不发布或部署。

## 6. 验证、审查与修正

执行与风险匹配的 focused tests、仓库质量门禁和必要的真实环境验收。不能执行的项目必须说明原因和剩余风险。

随后进行两轮审查：

1. **目标审查**：对照 TODO、计划边界、验收口径和非目标，检查漏做、越界和假完成。
2. **工程审查**：检查完整 diff、正确性、失败路径、并发、数据安全、权限、兼容、回归、测试质量和无关改动。

发现问题后持续执行“修正 → 受影响测试 → 重新审查”。需要新的产品决策或明显扩大范围时停止确认。

## 7. 提交并收口

只有同时满足以下条件才提交：

- 目标和验收口径已满足。
- 适用门禁通过，未执行项已有真实说明。
- 两轮审查没有未解决的必要问题。
- diff 只包含当前任务。
- Obsidian 中的 TODO 和计划状态已经更新：完成项从 TODO 删除，详细计划按需整理为完成摘要。

只暂存当前任务文件并使用仓库要求的提交格式。Obsidian 工作区不属于仓库提交范围。

用户明确要求完整本地收口且仓库规则允许时，按规则同步开发集成分支、重新验证并安全合并；不要因此自动推送远端。

## 8. 完成 Goal

仅在实现、验证、审查、Obsidian 工作流文档、Git 提交以及明确要求的本地收口全部完成后，将 Goal 标记为完成。

最终报告实际改动、验证结果、Git 状态、未验证项和剩余风险，然后停止。
