# 用户级智能体协作规则

本文件是 Kimi Code 用户级约束，适用于所有项目。项目目录中的 `AGENTS.md` 可以补充项目规则；发生冲突时，系统级规则优先，其次遵守更具体、更安全的约束。

## 简体中文强制规则

以下内容必须使用简体中文：

- AI 与用户的所有对话回复。
- 设计文档、API 文档、README、规范文档、任务规划文档。
- 代码注释，包括单行注释、多行注释和文档注释。
- Git 提交信息。
- 操作日志、coding log、review report、审查报告。
- 错误提示、警告信息、测试用例描述。
- 配置文件中的说明性文本。

唯一例外：代码标识符、变量名、函数名、类名、包名、接口路径、配置键、协议字段和第三方专有名称，遵循项目既有命名约定。

## Kimi Code 操作边界

- 回复应直接、简洁，先给结论；非简单任务开始前用一句话说明下一步。
- 修改前先读取相关代码、配置、日志和工作流文档，不臆测依赖、路径或运行状态。
- 坚持最小充分改动，不顺手重构、不格式化无关代码、不引入 speculative 抽象。
- 未经用户针对当前操作明确授权，不执行 `git commit`、`git push`、`git reset`、`git rebase`、强制推送、创建 Pull Request、标签、Release 或部署；Git 状态改变前必须再次确认具体动作。
- 删除文件、杀进程、覆盖未提交改动、修改共享数据或其他难以恢复的操作，必须先核对精确目标并获得确认。
- 不读取或输出不属于任务所需的秘密；不把密钥、令牌、密码、本机绝对路径、私有资产或运行日志提交到仓库。
- 改动完成后运行与风险匹配的测试、typecheck、lint 或真实环境检查；无法验证的部分必须明确说明。

## 项目工作区

个人开发计划统一存放在 Obsidian，不再在仓库中创建或使用 `.agent/`。

每次处理项目任务时，按以下方式定位该项目的工作区：

1. 用 Git 根目录作为项目根目录；不是 Git 仓库时使用当前工作目录。
2. 优先读取 `AGENT_PROJECT_WORKFLOW_CONFIG` 指定的文件；未设置时依次尝试 `~/.config/agent-project-workflow/config.toml` 和兼容旧安装的 `~/.codex/project-workflow.toml`。
3. 使用 `[projects]` 中与项目根目录绝对路径完全匹配的 Vault 内相对目录；每个项目都必须有精确映射。
4. 最终工作区为 `<vault_root>/<项目映射>/`。

配置文件缺失、`vault_root` 不可用、项目尚无映射或映射有歧义时，使用 `$agent-dev-workflow-init`：首次初始化必须让用户选择 Obsidian Vault 和 Vault 内的完整项目路径，不得自动放入 `Myproject`。后续改变位置也必须通过该 Skill 预览并安全改绑或迁移，不得只移动目录或只改配置。旧配置中的 `projects_root` 仅用于识别待迁移的历史默认位置，不再作为新项目回退规则。

项目工作区的常用结构如下：

- `TODO.md`：当前仍未完成的跨计划任务。
- `planning/计划中/`：已经立项、尚未开始执行的方案。
- `planning/执行中/`：正在实施或验收的方案。
- `planning/已完成/`：完成摘要。
- `constraints/`：长期约束和架构边界。
- `explanations/`：产品、需求和设计说明。
- `handoff/`：需要在不同机器或不同人员之间交接的材料。
- `artifacts/`：需要随项目文档长期保留的小型图像或验证材料，不放构建缓存和运行日志。

项目工作区可以由 OneDrive 或其他同步盘同步，可以包含个人计划和私有上下文，但不得存放 API Key、令牌、密码、真实用户数据或其他秘密。机器本地的开发秘密放在 `~/.config/agent-project-workflow/local/<仓库名>/`，文件权限必须限制为当前用户；能够使用系统凭据库时优先使用系统凭据库。

Obsidian 工作区是个人开发事实源，不写入仓库，也不应在项目的公开 `README.md`、`AGENTS.md` 或 `CLAUDE.md` 中暴露本机路径和个人计划。需要外发的内容必须先去敏、重整后再进入公共文档。

## 代码任务强制调查门禁

代码任务必须先调查真实现场，再判断是否执行。不得看到需求、方案或计划后直接改代码。

### 适用范围

用户提出以下任一任务时，必须先进入调查阶段：

- Bug 修复、运行异常、报错分析、线上或本地故障排查。
- 新功能需求、功能改造、代码重构、架构调整、性能问题。
- UI 调整、交互优化、视觉修改、文案调整。
- 用户提供外部方案、计划文档、截图、日志或调研材料。
- 文档中包含待确认的产品选择、兼容策略、影响范围或验收口径。

### 调查阶段必须完成的事

- 读取真实仓库状态、相关源码、配置、日志、数据库或服务状态。
- 读取项目工作区中的 TODO、相关计划、约束和说明文档。
- 判断用户真实意图、影响范围、风险点、验收口径和是否存在待确认问题。
- 对前端、重构、架构和业务流程先形成结构理解，不能拍脑袋直接实现。

### 调查阶段允许做的事

未获得明确实现授权前，只允许：

- 使用 `rg`、`git status`、`git diff`、日志查看和配置查看等方式调查现状。
- 编写或更新项目工作区中的计划、TODO、约束和说明文档。
- 向用户汇报调查结论、风险、待确认问题和下一步建议。

### 调查阶段禁止做的事

未获得明确实现授权前，禁止：

- 修改业务代码、前端页面、后端逻辑、配置文件、SQL、脚本或公开文档。
- 运行会改变项目状态的格式化、生成、迁移、安装或构建副作用命令。
- 把临时旁路验证结果描述成用户当前真实运行实例已经验收通过。
- 跳过项目工作区中的计划和 TODO，直接按外部方案落地实现。

### 实现授权判定

“开始实现”“按这个计划改代码”“直接落地”“执行这个方案”“修复它”“实现这个功能”“继续执行”等表达属于明确授权。

“看看”“分析一下”“调查一下”“整理方案”“写 TODO”“评估一下”“这个计划怎么样”，以及仅提供材料，默认不是实现授权。

正确流程是：先调查真实现场；需求清晰且用户已授权时继续实现；关键选择尚未确认时，先写入项目工作区并向用户确认。

## 工作流文档规则

- 计划文档是实施依据，不是任务看板；具体待办统一写入 `TODO.md`。
- 计划说明背景、目标、边界、方案、影响范围、验证口径、风险和失效条件，不维护每日流水。
- 完成事项从 `TODO.md` 删除；有长期价值的结果写入完成摘要、约束或说明。
- 废弃方案直接删除；有价值的结论先提炼到仍然有效的文档。
- 同一内容不在 Obsidian 工作区和仓库公共文档中重复维护。

## 代码编写规范

- 必要时编写规范注释，让技术人员理解复杂边界、设计原因和风险。
- 注释只解释复杂逻辑、设计意图或风险，不写空泛重复内容。
- 新增实现优先复用项目既有模块、helper、hook、组件和公共能力。
- 新增抽象必须能减少真实复杂度、降低耦合或匹配项目既有架构。
- 业务逻辑结构、职责边界、可维护性、可解释性和可靠性优先于临时糊一个能跑的版本。
