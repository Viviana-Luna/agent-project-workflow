# 客户端接入

## Codex

- 用户级规则：`~/.codex/AGENTS.md`
- 推荐共享 Skill：`~/.agents/skills/`
- 也可以使用 Codex 专属 Skill 目录：`~/.codex/skills/`

新交互安装流程将工作流 Skill 作为独立内容安装到共享目录 `~/.agents/skills/`，并诊断旧 Codex 专属目录中的同名 Skill。

选择共享 Skill 时，管理器按同名目录的完整内容哈希显示“未安装”“已是当前版本”或“版本有差异”。用户可逐项多选，或按 `A` 全选后只覆盖所选 Skill；未托管或损坏目录仍须走归档或明确的无备份替换流程。

共享 Skill 安装完成后，交互向导会继续让用户选择一个已检测客户端的规则，或选择跳过。Skill 与规则分别预览、分别确认，不会因检测到客户端而自动写入规则。

客户端接入不选择 Obsidian Vault，也不创建项目映射。项目文档位置只在首次调用 `$agent-dev-workflow-init` 时由用户选择，后续仍通过该 Skill 安全调整。

## Claude Code

- 用户级规则：`~/.claude/CLAUDE.md`
- 用户级 Skill：`~/.claude/skills/`

Claude Code 不与其他三个客户端共享用户级 Skill 目标。新交互流程只单独安装规则；Claude 专属 Skill 仍由兼容批量入口维护，避免对 OpenCode 造成重复发现。

## Kimi Code

- 用户级规则：`$KIMI_CODE_HOME/AGENTS.md`；未设置时为 `~/.kimi-code/AGENTS.md`
- 用户级 Skill：`~/.agents/skills/` 或 `$KIMI_CODE_HOME/skills/`
- 新规则和 Skill 通常在新会话中加载。

## OpenCode

- 用户级规则：`~/.config/opencode/AGENTS.md`（macOS/Linux/Windows 一致；`~` 在 Windows 展开为 `%USERPROFILE%`）
- 用户级 Skill：`~/.agents/skills/`

OpenCode 与 Codex、Kimi Code 共用 Agent Skills 兼容目录，管理器会合并所有权，不重复复制同一 Skill。

> 已核实 opencode 1.18.4 在 Windows（`XDG_CONFIG_HOME` 未设置）下配置目录仍为 `~/.config/opencode`（`opencode.json` 的 `instructions` 指向同目录 `AGENTS.md`）；`%APPDATA%\opencode` 仅存放 WebView2 运行时缓存，不是配置目录。

## 平台说明

- macOS/Linux 上 `~` 展开为用户主目录；Windows 上 `~` 展开为 `%USERPROFILE%`。Codex、Claude Code、Kimi Code、OpenCode 的规则目录在三个平台一致（`~/.codex`、`~/.claude`、`~/.kimi-code`、`~/.config/opencode`）。
- 客户端差异由 `adapter.toml` 的 `rule_target` 声明（必要时可用可选的 `rule_target_windows` 覆盖 Windows 路径），核心代码按 `os.name` 选择，不在多处散落路径判断。

## 托管规则区块

管理器不再默认覆盖整份用户规则，而是在目标文件中维护以下区块：

```markdown
<!-- agent-project-workflow:start -->
工作流规则
<!-- agent-project-workflow:end -->
```

区块外内容归用户所有。管理器只比较托管区块的内容版本，显示“未安装”“已是当前版本”或“版本有差异”。旧规则没有托管标记、标记损坏或文件无法读取时，必须先查看差异，再选择压缩归档后替换或无备份直接替换。

## 适配原则

- 不在公共适配器中写入真实 vault 路径。
- 客户端规则只描述如何查找通用配置和工作区。
- 客户端专属能力放在对应适配器，不污染核心目录规范。
- 安装前先预览；已有用户文件默认不覆盖。
- 客户端差异由 `adapter.toml` 声明，新增客户端不得把路径判断散落到工作流核心。
